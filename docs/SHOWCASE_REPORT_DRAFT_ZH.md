## 1. 提出问题：为什么现在需要新的文献调研方式

### 1.1 背景

近年来，人工智能领域发展极快，模型、方法和应用几乎以周为单位快速更新。新的预训练方法、多模态系统、具身智能路线、Agent 框架、推理增强方法等不断涌现，带来的直接结果就是：

- 新论文数量持续激增
- 顶会和期刊投稿量快速上升
- 同一问题往往在很短时间内就出现多种不同路线
- 研究热点变化快，已有调研结果很快会过时

尤其在人工智能相关方向中，研究者面对的不是“找不到文献”，而是“文献太多、变化太快、结构太复杂”。

### 1.2 传统文献调研方式的痛点

在传统流程中，研究者通常需要：

- 自己构造关键词
- 在不同数据库和搜索引擎中反复检索
- 打开大量论文页面逐篇筛选
- 用笔记工具手动归纳研究方向
- 自己维护阅读顺序和比较关系
- 在需要的时候再把内容整理成汇报或综述材料

这种方式存在几个明显问题：

1. 时间和精力消耗非常大  
研究者会把大量时间花在查找、筛选、整理和重复记录上，而不是花在真正的判断和思考上。

2. 调研过程容易中断  
一次检索做完以后，如果几天后想继续，往往要重新回忆上次做到哪里、看过哪些论文、比较过哪些路线。

3. 研究结构不容易沉淀  
论文、方向、候选方案、实验设置、对比结论，常常分散在浏览器、PDF、笔记和聊天记录里，难以形成统一的研究工作台。

4. 多轮调研容易重复  
不同时间做出来的检索结果会有重合，用户很难保证新一轮调研既能继承旧进度，又不被重复文献淹没。

5. 过程可解释性弱  
很多时候用户最后能得到一个结论，但无法清晰追踪“为什么会得到这个结论”“中间经历了哪些筛选与分支”。

一句话概括：

> 当前文献调研的核心矛盾，不是信息不足，而是信息过载条件下缺少一个能够持续推进、持续组织、持续继承进度的研究工作台。

## 2. 现状：已经有 AI 文献调研工具，但仍存在关键缺口

### 2.1 当前已经有哪些工具

目前市场上已经出现了很多和文献调研相关的工具，例如：

- ChatGPT Deep Research
- Gemini Deep Research
- Elicit
- ResearchRabbit
- Connected Papers
- Scite
- Zotero

这些工具分别在不同环节提供了帮助：

- 有些擅长一次性生成调研报告
- 有些擅长结构化提取 evidence
- 有些擅长做文献发现和图谱可视化
- 有些擅长管理文献库、标签和引用

所以问题并不是“完全没有工具”，而是现有工具大多只强在某一个切面。

### 2.2 这些工具已经缓解了哪些问题

现有 AI 工具确实带来了明显改进：

- 提高了初始检索速度
- 降低了整理摘要的门槛
- 可以帮助用户快速进入一个陌生方向
- 可以自动生成初步综述或结构化总结
- 某些工具已经支持先出计划、再运行研究、最后生成报告

因此，今天如果只问“AI 能不能帮忙调研文献”，答案显然是可以。

### 2.3 但现有主流方案仍存在一个核心问题

尽管这些工具已经很强，但大多数方案的底层交互逻辑仍然更接近：

**一次指令 -> 一次运行 -> 一份结果**

这会带来几个连续研究中的实际问题：

1. 后续跟进成本高  
如果用户想继续上一轮调研，往往需要重新组织整套提示词，重新告诉系统背景、限制、目标和重点。

2. 研究进度继承不稳定  
虽然已有工具可以保留一定的上下文，但它们更偏向“保留对话上下文”，而不是“保留一套持续研究状态”。

3. 两次调研之间容易断层  
用户可能在第一轮得到了一个不错的结果，但第二轮想沿着其中某个分支继续深入时，系统并不天然具备“分支继承”“节点继续”“集合派生”的工作机制。

4. 文献重复与结果漂移  
多轮调研很容易出现重复检索、重复总结、结果不连贯、文献范围重新漂移的问题。

5. 缺少真正的工作台结构  
很多工具最终还是围绕聊天界面或报告结果组织，难以把 project、task、collection、graph、notes、assets 串成统一的研究环境。

因此，我们认为现有主流工具的不足，不是“不会调研”，而是：

> 对单次任务的结果支持已经很强，但对长期研究过程的进度继承、结构组织和分支推进支持仍然不足。

## 3. 我们的方法：构建一个可持续推进的本地研究工作台

### 3.1 我们的目标

针对前面的问题，我们的目标不是再做一个“聊天式调研助手”，而是构建一个：

- 面向长期研究过程
- 可中途调整方向
- 可保留研究进度
- 可组织论文集合
- 可追踪运行过程
- 可输出阶段性结果
- 可和现有工具协同

的本地研究工作台。

### 3.2 我们提出了什么

我们提出并实现的是一个 `research-local` 研究系统，当前项目名称为：

**ResearchMate**

它的核心形态不是单轮问答，而是一套状态化研究环境，围绕以下对象组织：

- `project`：顶层研究分组
- `task`：具体研究任务
- `collection`：可复用的论文集合
- `paper`：论文节点与资产
- `canvas`：用户研究工作台
- `run events`：运行事件流
- `artifact`：阶段报告、导出文件、可视资产

### 3.3 系统的核心设计

#### 设计一：把单次调研变成可持续推进的任务系统

用户不是只给一次提示词，而是可以持续维护：

- 当前项目
- 当前任务
- 当前方向
- 当前轮次
- 当前论文集合
- 当前研究图谱

这就使研究进度成为系统的一等公民，而不是一次性结果的附属物。

#### 设计二：支持两种研究模式

系统不是只支持一种使用方式，而是同时支持：

1. `GPT Step`
   - 用户主导
   - 一步一步推进
   - 每一步都能基于当前结果再决定下一步

2. `OpenClaw Auto`
   - 代理自治探索
   - 自动构建阶段性结果
   - 在 checkpoint 时暂停，等待用户 guidance 后继续

这样既能满足“我要自己控制研究节奏”的需求，也能满足“我希望系统先自动跑起来”的需求。

#### 设计三：将研究结果层和工作台层分离

系统把图谱分成两层：

- `canonical research graph`
- `canvas state`

前者代表研究结果本身，后者代表用户自己的工作台组织方式。

这使得用户可以：

- 拖拽节点
- 调整排版
- 增加手工节点
- 添加手工边
- 写备注
- 隐藏系统节点

同时又不会破坏系统本身生成的研究结构。

#### 设计四：把论文管理与研究执行接起来

系统支持：

- Zotero 本地导入导出
- collection 持续复用
- 从 collection 创建 study task
- 对选中的论文集合继续总结、比较和派生研究

这样一来，Zotero 不再只是文献仓库，而可以自然接到研究执行工作流中。

#### 设计五：保留研究过程的可解释性

OpenClaw Auto 不只是返回最终结论，而是保留过程性事件：

- progress
- node_upsert
- edge_upsert
- paper_upsert
- checkpoint
- report_chunk
- artifact

这让用户不仅能看到结果，也能看到过程，并在关键阶段接管方向。

### 3.4 我们的方法缓解了什么问题

基于上述设计，这个系统主要缓解了以下问题：

1. 缓解了多轮调研时的进度断裂问题  
研究可以在上一次任务基础上继续，而不必从零重建上下文。

2. 缓解了提示词重复构造问题  
用户不需要每次都重新描述完整背景，因为 project、task、collection 已经保存了状态。

3. 缓解了结果重复和文献重复问题  
研究对象不再只是临时对话，而是可复用、可组织、可比较的结构化对象。

4. 缓解了研究结构难以沉淀的问题  
用户可以把方向、论文、比较结论、阶段报告和导出文件放在同一工作台里管理。

5. 缓解了黑盒代理不可解释的问题  
自治研究过程通过事件流和 checkpoint 暴露出来，便于观察、引导和继续推进。

## 4. 实际实现：系统是如何落地的

### 4.1 运行形态

当前系统默认运行在：

- `WSL / Linux VM`
- 本地单用户模式
- `research_local` profile

主要由三部分组成：

- `backend`
- `worker`
- `frontend`

当前已经形成一个可运行的 research-only 主线，而不再依赖旧的企业微信、提醒、移动端认证等链路。

### 4.2 后端能力

后端当前已经支持：

- research task 创建与读取
- GPT Step 链路
- OpenClaw Auto 链路
- canvas 状态存储
- run events 事件流
- node chat
- paper asset
- collection 和 study task
- Zotero 本地导入导出

从系统结构上看，它已经不是一个单脚本式原型，而是一套具备明确对象模型的研究系统。

### 4.3 前端工作台

前端当前采用独立工程实现，主要特点是：

- 全屏工作台
- 左中右三栏布局
- 中间卡片式画布
- 右侧详情 / 对话 / PDF / Fulltext / 时间线
- 左侧 project / task / collection 组织层

和单轮聊天相比，这种界面更贴近真实研究过程，因为研究并不只是“问一句，答一句”，而是同时需要：

- 看结构
- 看节点关系
- 比较论文
- 管理集合
- 查看阶段结果

### 4.4 两种研究模式的实际落地

#### GPT Step

当前已经打通的典型链路包括：

- 创建任务
- 规划方向
- 检索方向
- 开始探索
- 生成候选
- 选择候选
- 继续下一轮
- 构建图谱
- 全文处理
- 论文总结

这条链路适合强调“用户主导”的研究方式。

#### OpenClaw Auto

当前已经打通的典型链路包括：

- start
- progress
- checkpoint
- guidance
- continue
- report
- artifact

这条链路适合强调“自治探索 + 人类阶段引导”的研究方式。

### 4.5 当前 Demo 能力

为了方便展示，当前项目已经准备了两种 demo 形式：

1. 静态展示 Demo  
围绕“具身智能 / Embodied AI”主题，预置项目、任务、collection、画布、checkpoint、报告、资产和导出记录，用于直接展示完整研究结果。

2. 动态运行 Demo  
保留真实链路脚本，可以现场跑：

- `gpt_basic`
- `gpt_explore`
- `openclaw_auto`

这意味着展示时既能给出“已经跑好的最终效果”，也能演示“系统是如何一步一步运行起来的”。

推荐按以下顺序演示：

1. 先展示项目整体界面  
展示 project、task、collection、canvas、detail panel 的整体布局，让听众先建立“这不是一个普通聊天工具”的直观印象。

   - 方向结构
   - checkpoint
   - 报告与资产
   - PDF / Fulltext

3. 展示 GPT Step 流程  
从新任务开始，依次点击：
   - 创建任务
   - 规划方向
   - 检索方向
   - 开始探索

现场讲解时，可以这样概括：

> 这里我先不把它看作一个 AI 聊天工具，而把它看作一个研究工作台。左侧是研究对象的组织层，中间是研究图谱和工作画布，右侧是当前节点、资产和运行过程。这样我们不是在反复发起一次性提问，而是在一个持续环境里推进研究。

## 6. 我们的方法相比现有工具的优势

### 6.1 和 Deep Research 类产品相比

相较于 ChatGPT / Gemini Deep Research 这类产品，我们的区别不在于“它们完全不能调整”，而在于：

- 它们整体更偏向单次任务和单份结果
- 我们更强调长期研究过程和状态继承

所以我们的优势是：

- 更适合持续调研
- 更适合多轮推进
- 更适合沿着分支继续探索
- 更适合保存研究结构

### 6.2 和文献发现工具相比

相较于 ResearchRabbit、Connected Papers 这类工具，我们不只做：

- 文献发现
- 图谱浏览

而是继续承接：

- 任务推进
- 论文比较
- 阶段总结
- 资产沉淀
- 后续导出

### 6.3 和文献管理工具相比

相较于 Zotero，我们不只做：

- 收藏
- 标签
- 同步
- 引用

而是把 collection 继续接到：

- summary
- compare
- graph
- study task
- 研究工作流

### 6.4 最值得强调的差异化优势

如果最后要凝练成几句最重要的话，我们建议这样表述：

1. 这是一个持续研究系统，而不是一次性报告工具。
2. 它支持人类主导和代理自治两种协作模式。
3. 它把文献发现、研究推进和结果沉淀放进同一工作台。
4. 它和现有文献生态协同，而不是要求用户完全改用新流程。
5. 它既重视结果，也重视研究过程的可解释性和可继承性。

## 7. 总结

### 7.1 本项目的核心贡献

本项目的核心贡献，不在于单纯再做一个 AI 问答工具，而在于：

- 把论文调研从“一次性查询”转变成“持续研究过程”
- 把 AI 从“单次回答者”转变成“研究协作者”
- 把文献调研从“分散在多个工具中的碎片流程”转变成“统一的本地研究工作台”

### 7.2 项目的核心价值

我们认为，这个项目真正有价值的地方在于：

1. 它解决的是研究过程组织问题，而不仅仅是检索速度问题。
2. 它强调长期进度继承，而不仅仅是一次回答质量。
3. 它同时兼顾用户控制权和代理自治能力。
4. 它适合继续扩展成真正可持续使用的研究系统。

### 7.3 最后一句总结

可以用下面这段作为汇报结尾：

> 总的来说，我们做的不是一个只会输出答案的工具，而是一个能够持续承接研究进度、支持中途改向、支持集合组织、支持自治探索、并且能够将研究过程和结果一起沉淀下来的本地研究工作台。它更适合真实研究者在长期课题中的实际工作方式，这也是它相对于现有文献调研工具最重要的差异化价值。
