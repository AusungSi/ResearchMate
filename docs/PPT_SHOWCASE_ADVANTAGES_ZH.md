# ResearchMate 展示优势整理

更新时间：2026-04-22

这份文档面向项目展示、路演、答辩和 PPT 制作。
目标不是展开所有技术细节，而是把“这个项目为什么值得做、为什么和现有工具不同、为什么它有继续发展的价值”讲清楚。

## 1. 一句话定位

`ResearchMate` 不是一个一次性生成报告的 AI 文献助手，而是一个面向长期研究过程的本地研究工作台：把人类主导的 step-by-step 调研、自治代理探索、文献管理迁移、图谱组织和阶段性总结整合到同一个可持续推进的研究环境中。

## 2. 最值得先讲清楚的 3 个核心优势

### 2.1 持续研究，而不是一次性出报告

和主流 `Deep Research` 类产品相比，这个项目更像“研究工作台”，而不只是“研究结果生成器”。

它的重点不是让模型一次性写完一份报告，而是把研究过程拆成：

- 可创建的任务
- 可持续推进的方向
- 可中途调整的探索分支
- 可回看的节点、图谱、日志、报告和资产

这意味着用户可以在研究中途重新改方向、补约束、扩分支、继续某个子任务，而不是每次都重新发起一次全新的研究请求。

### 2.2 不是孤立对话，而是可迁移的研究工作流

很多 AI 工具的问题不是“不会回答”，而是“进度留不住、结构不可复用、和现有工具脱节”。

这个项目的优势是把研究对象组织成一套持续状态：

- `project`
- `task`
- `collection`
- `paper`
- `canvas`
- `run events`
- `artifact`
- `export`

同时又能和现有生态协同：

- 接入 Zotero
- 支持本地导入导出
- 支持 API 研究链路
- 支持研究结果继续迁移到外部工具

所以它不是替代文献管理工具，而是把“文献管理之后真正的研究过程”接起来。

### 2.3 双模式研究引擎：可控，也可自治

当前项目不是只有一种研究方式，而是明确支持两条路线：

- `GPT Step`
  - 用户主导
  - 逐步推进
  - 每一步都由用户决定是否继续
- `OpenClaw Auto`
  - 代理自治探索
  - 自动构建阶段性结果
  - 在 `checkpoint` 等待用户引导后继续

这让系统同时覆盖两类典型需求：

- “我想自己掌控研究节奏”
- “我想先让系统跑起来，再在关键节点接管”

这一点比单一模式工具更适合真实研究场景。

## 3. 可重点展示的完整优势清单

下面这些点，都是适合写进 PPT 的“产品价值点”。

### 3.1 人机共研，而不是黑盒代做

这个项目不是把用户排除在外，而是把用户放在研究流程中：

- 用户可以规划方向
- 可以检索某个方向
- 可以选择候选
- 可以继续某一轮探索
- 可以在 checkpoint 时给出 guidance

系统做的是“放大研究能力”，而不是把研究过程彻底黑盒化。

### 3.2 研究过程是状态化、可回放、可追踪的

项目会保留：

- 任务状态
- 节点与边
- 运行事件流
- 阶段报告
- 资产文件
- 节点问答
- 导出历史

这意味着研究不是一次性的，而是“可以回看、可以继续、可以复盘”的。

### 3.3 研究图谱和用户工作台分层

这是当前架构里很重要、也很适合对外展示的一点：

- `canonical graph` 代表系统沉淀的研究结构
- `canvas state` 代表用户自己的工作台组织方式

用户可以自由拖拽、隐藏、加注释、加手工节点、加手工边，而不会破坏系统研究结果本身。

这个分层设计很适合长期研究，因为它同时保留了：

- 研究事实层
- 认知组织层

### 3.4 不只支持“对单篇论文理解”，也支持“围绕方向和集合继续研究”

系统不是只读一篇 paper 就结束，而是支持：

- 从方向继续探索
- 从 round 继续推进
- 从 collection 派生新的 study task
- 多篇 paper compare 后继续研究

这意味着它天然更适合中大型研究任务，而不是只适合“快速问一个问题”。

### 3.5 从文献发现到结构化输出形成闭环

当前工作台已经把多种能力串起来：

- 文献发现
- 方向规划
- 论文检索
- 候选方向生成
- 图谱构建
- 全文处理
- 论文总结
- 比较分析
- 阶段报告
- 导出

相比“只负责找论文”或者“只负责写综述”的工具，它更接近一个完整的研究执行台。

### 3.6 兼容现有研究生态，迁移成本低

这个项目不是要求用户推倒重来，而是强调兼容：

- Zotero 本地导入导出
- API 调用链路
- 外部 provider 接入
- 后续可接更多发现源和 citation 源

这使它更容易进入真实使用场景，因为用户通常已经有自己的文献管理工具和既有流程。

### 3.7 本地部署带来更高的可控性

当前主线以 `research_local + WSL / Linux VM` 为默认形态，这带来几类实际优势：

- 数据流向更可控
- 研究产物保留在本地
- API key、自定义 provider、更换模型都更灵活
- 更适合后续做深度定制

这对于重视隐私、稳定性和可扩展性的用户尤其重要。

### 3.8 OpenClaw Auto 的事件流设计让自治过程“能讲清楚”

很多 agent 工具的问题是：它跑了，但用户不知道它在干什么。

这个项目的 `OpenClaw Auto` 不是直接只给最终结果，而是通过：

- `progress`
- `node_upsert`
- `edge_upsert`
- `paper_upsert`
- `checkpoint`
- `report_chunk`
- `artifact`

把过程拆成可观察、可解释、可引导的阶段。

这对于研究场景尤其重要，因为研究用户往往不只要答案，还要知道系统是怎样得到这个答案的。

### 3.9 前端工作台更接近真实研究，而不是单轮聊天

当前前端不是单纯聊天框，而是：

- 左侧组织层
- 中间研究画布
- 右侧详情 / 对话 / 资产 / 时间线

这种结构更贴近真实研究者的工作方式，因为研究通常不是连续聊天，而是：

- 查看结构
- 比较材料
- 组织节点
- 追踪进度
- 在需要时发起针对性提问

### 3.10 具备演示友好性

项目现在已经不仅能跑，还具备较好的展示结构：

- 静态展示 Demo
- 动态运行 Demo
- 任务、collection、checkpoint、artifact、PDF/fulltext、导出记录都可以演示

这使它不仅适合作为工程项目，也适合作为课程答辩、作品集项目、研究型产品原型来展示。

## 4. 与市面主流工具的对比

下面这部分建议直接做成 PPT 表格。

### 4.1 和 ChatGPT / Gemini Deep Research 的对比

这类产品的强项是：

- 上手快
- 结果成文快
- 面向通用信息调研体验成熟
- 现在也支持先出计划、查看进度、再生成报告

但它们整体仍然更偏：

- 一次任务
- 一份结果
- 以报告为中心

本项目相对更强的地方：

- 更强调“长期研究过程”，而不只是一次研究结果
- 支持用户在过程中持续改向和继续分支
- 研究结构会沉淀为 `project / task / collection / canvas / artifact`
- 不是只保留聊天记录，而是保留可继续推进的研究状态

一句话对比：

> Deep Research 更像“研究报告生成器”，本项目更像“研究过程工作台”。

### 4.2 和 Elicit 的对比

`Elicit` 的强项是：

- 结构化 evidence synthesis
- systematic review
- screening / extraction / summary table
- 适合偏证据提取和系统综述的流程

本项目相对更强的地方：

- 更强调开放式探索，而不是只强调综述提取
- 更适合研究问题还在形成中的阶段
- 更强调图谱、方向、节点、分支和长期项目组织
- 支持自治代理探索模式，而不是只做 structured review

一句话对比：

> Elicit 更强在“结构化证据提取”，本项目更强在“开放式研究探索 + 长期工作台组织”。

### 4.3 和 ResearchRabbit / Connected Papers 的对比

这类工具的强项是：

- 文献发现体验好
- 图谱和相似关系可视化强
- 容易从一篇 paper 出发不断扩展阅读

本项目相对更强的地方：

- 不只做发现，还继续承接后续研究执行
- 不只显示图，还能把图谱变成任务、总结、比较、导出和后续分支
- 支持 step-by-step 和自治代理两种研究推进模式
- 具备 project / task / collection / artifact 这套更完整的研究组织层

一句话对比：

> 它们更像“发现工具”，本项目更像“发现 + 组织 + 推进 + 产出”的一体化研究工作台。

### 4.4 和 Scite 的对比

`Scite` 的强项是：

- Smart Citations
- 支持 / 反驳 / 提及 的证据判断
- 强调答案可验证、证据可追溯

本项目相对更强的地方：

- 更强调完整研究流程，而不是只强调 citation evidence
- 可把发现、比较、分支探索、图谱组织放在同一工作台内
- 更容易结合用户自己的研究结构长期沉淀

一句话对比：

> Scite 更强在“证据判断”，本项目更强在“把研究从发现一路推进到结构化产出”。

### 4.5 和 Zotero 的对比

`Zotero` 的强项是：

- 文献管理
- 收藏、标签、同步、引用
- 适合做文献库管理的基础设施

本项目相对更强的地方：

- 不停留在“存文献”
- 可以把 collection 继续变成 study task
- 可以继续做 compare、summary、graph、fulltext、artifact
- 可以把管理层直接接到研究执行层

一句话对比：

> Zotero 更像“文献仓库”，本项目更像“基于文献仓库继续推进研究”的工作台。

## 5. 适合强调的差异化结论

如果你在 PPT 里只能留下 5 条最核心的记忆点，建议用这 5 句：

1. 它不是一次性出报告，而是支持长期推进的研究工作台。
2. 它把“人类主导”和“代理自治”放进同一套研究系统里。
3. 它不只是查文献，而是把发现、组织、比较、总结和导出串成闭环。
4. 它能和 Zotero、本地文件、API 链路协同，不要求用户推翻原有流程。
5. 它把研究结果层和用户工作台层分开，既保留研究结构，也保留个人组织自由度。

## 7. 建议的 PPT 展示顺序

如果你后面要做一套 8 到 12 页的展示，建议按这个顺序来：

### 第 1 页：项目定位

- 这是一个本地研究工作台
- 不是一次性报告工具
- 目标是支持长期研究过程

### 第 2 页：问题背景

- 现有工具大多只解决某一段流程
- 报告型工具缺少长期状态
- 文献管理工具缺少研究推进能力
- 图谱工具缺少任务化与产出化

### 第 3 页：三个核心优势

- 持续研究
- 可迁移工作流
- 双模式研究引擎

### 第 4 页：工作流闭环

- project
- task
- collection
- canvas
- compare
- artifact
- export

### 第 5 页：GPT Step

- 用户主导
- step-by-step
- 中途可调整

### 第 6 页：OpenClaw Auto

- 代理自治探索
- checkpoint 引导
- 阶段性结果沉淀

### 第 7 页：与现有工具对比

- ChatGPT / Gemini Deep Research
- Elicit
- ResearchRabbit / Connected Papers
- Scite
- Zotero

### 第 8 页：前端工作台

- 三栏结构
- 图谱和画布
- 节点详情
- PDF / Fulltext / Timeline

### 第 9 页：演示能力

- 静态 demo
- 动态 demo
- 可展示完整研究链路

### 第 10 页：总结

- 这个项目的价值不在于“再做一个聊天工具”
- 它的价值在于把研究过程本身产品化

## 8. 适合答辩时说的总结段

可以直接用下面这段作为口头总结：

> 这个项目的核心价值，不是单纯让模型帮我们写一份文献综述，而是把研究过程本身变成一个可以持续推进、可以中途调整、可以沉淀结构、可以和现有文献生态协同的工作台。和主流的一次性 Deep Research 工具相比，它更强调长期状态；和 Zotero、ResearchRabbit、Scite 这类单点能力工具相比，它更强调把发现、组织、推进和产出串成闭环。这也是它最值得继续做下去的地方。
